# Alpauls Recruitment

A recruitment management application with role-based access control, task tracking, and team collaboration.

## Language

**Role**:
A named set of permissions assigned to a User. Predefined roles are coded in TypeScript; the `"custom"` role carries per-user permission overrides.
_Avoid_: Permission group, access level, user type

**Custom Role**:
A per-user permission set stored as a JSON column (`permissions`) on the User table. When a User's role is `"custom"`, their permissions are read from this column instead of from the predefined role definition. Not a reusable role template — each custom-user has independent permissions.
_Avoid_: Dynamic role, flexible role, ad-hoc role

**Resource**:
A permission domain in the access control statement (e.g., `clients`, `job_mandates`, `tasks`). Each resource defines the actions that can be granted.
_Avoid_: Entity, domain, module

**Action**:
A granular operation on a Resource (e.g., `create`, `read`, `assign`). Roles map Resources to lists of allowed Actions.
_Avoid_: Permission, operation, capability

**Admin**:
A User with the `admin` role that bypasses all permission checks. Can create, update, and archive other Users, assign roles, and manage the system.
_Avoid_: Superuser, root, system user

**Task**:
A trackable unit of work with ownership, deadlines, and optional links to recruitment entities. Has a Creator (the User who made it), an Assignee, an optional start date and a deadline; relative Reminders are only valid when a deadline is set.
_Avoid_: Todo, ticket, item

**Task Link**:
A typed reference from a Task to another domain entity. Valid entity types: `prospect`, `job_mandate`, `client`, `client_contract`.
_Avoid_: Relation, connection, association

**Task Event**:
An immutable record of a change made to a Task (who changed what, when). Every edit produces a Task Event.
_Avoid_: Audit log, changelog, history entry

**Change Record**:
An immutable record of a change made to any entity (who changed what, when). Every entity (Client, Client Contract, Job Mandate, Prospect, Task) produces Change Records on every edit. Task Events are the existing implementation of this pattern.
_Avoid_: Audit log, changelog, history entry

**Assignee**:
The single User responsible for completing a Task.
_Avoid_: Owner, responsible

**Creator**:
The User who created a Task. Distinct from the Assignee — the Creator may assign the Task to someone else.
_Avoid_: Author, owner, reporter

**Task Status**:
The lifecycle stage of a Task: `todo`, `in_progress`, or `done`.
_Avoid_: State, progress, stage

**Reminder**:
A scheduled trigger at a concrete datetime (`trigger_at`). Reminders are either attached to a Task (with optional relative offset before the task's deadline) or standalone (absolute time only, no reference point). A Task may have multiple Reminders. When a Reminder fires, it may produce a Notification.
_Avoid_: Alarm, alert (use **Notification** for the delivered message)

**Notification**:
A delivered message to a User, generated by system events (task assignment, status change, approaching deadline, fired Reminder, or system announcement). Has a type, title, body, and read/unread state. Distinct from Reminder — a Reminder schedules the trigger; a Notification is the result delivered to the user.
_Avoid_: Alert, push message, toast

**Supervisor**:
A User who oversees one or more other Users (their Subordinates). The Supervisor relationship gates access — a Supervisor can manage and view data scoped to their Subordinates.
_Avoid_: Manager, boss, lead

**Subordinate**:
A User who reports to a Supervisor. A User has at most one Supervisor.
_Avoid_: Report, direct report, underling

**Client**:
A company or organization that the agency recruits for. The party paying for recruitment services. Has an Assignee (the BD responsible for the account).
_Avoid_: Customer, company, account

**Client Contract**:
A service agreement between the agency and a Client, defining the terms of engagement. Has an Assignee (initially the BD who created it; BD may assign an RM to manage it).
_Avoid_: Agreement, deal, contract

**Job Mandate**:
An open position a Client wants filled. The core unit of recruitment work — defines the role, requirements, and terms. Has an Assignee (the User responsible for filling it). The assignment chain: RM creates and assigns to TL; TL may re-assign to Callers. Lifecycle: `open` → `in_progress` → `hired` → `completed` (or `cancelled`). A mandate can have multiple positions with headcounts per location, stored as a JSON array: `[{ "location": "Dubai", "headcount": 3 }, { "location": "Mumbai", "headcount": 2 }]`. When all positions across all locations are filled, the mandate transitions to `hired`. Completed mandates are auto-archived.
_Avoid_: Job order, req, opening, vacancy

**Mandate Status**:
The lifecycle stage of a Job Mandate: `open`, `in_progress`, `hired`, `completed`, or `cancelled`.
_Avoid_: State, phase

**Prospect**:
Any person in the system, whether actively seeking a job or not. The person entity record. A Prospect's stage (prospect/applicant/candidate) is defined per Job Mandate — the same person can be at different stages for different mandates.
_Avoid_: Person, individual, lead

**Applicant**:
A stage — a Prospect who has expressed interest in a specific Job Mandate. Not a separate entity; the stage is an attribute of the Prospect↔JobMandate relationship.
_Avoid_: Job seeker, interested candidate

**Candidate**:
A stage — a Prospect who is QC-verified for a specific Job Mandate. Not a separate entity; the stage is an attribute of the Prospect↔JobMandate relationship.
_Avoid_: Verified prospect, qualified applicant

**Prospect Stage**:
The lifecycle of a Prospect relative to a Job Mandate: `prospect` → `applicant` → `candidate` → `hired` (or `rejected` at applicant stage). QC rejection sets stage to `rejected` with a reason (e.g., `internal_qc`). A Prospect with no mandate link is simply a Prospect.
_Avoid_: Status, phase, level

**Prospect Mandate**:
The relationship between a Prospect and a Job Mandate, storing the current stage (`prospect`, `applicant`, `candidate`, `hired`, `rejected`). A Prospect may have multiple Prospect Mandate records (one per linked mandate).
_Avoid_: Assignment, application, link

**Prospect Mandate Event**:
An immutable record of a stage change on a Prospect Mandate (who changed what stage, when, on which mandate). Tracks the full history of stage transitions per prospect↔mandate pair.
_Avoid_: Stage log, transition record

**SLA**:
A service-level agreement defining performance targets (e.g., time-to-fill, response time) for a Client Contract or Job Mandate. Stored as a static JSON config file — not a database entity and not access-controlled.
_Avoid_: OLA, KPI, target, metric

## Relationships

- A **User** has exactly one **Role** (one of: `admin`, `bd`, `rm`, `sc`, `tl`, `caller`, `qc`, `custom`)
- A **User** with Role `"custom"` has a `permissions` JSON column mirroring the access control statement shape
- The **Admin** role bypasses all permission checks — no explicit permissions needed
- Changing a **User's** Role or permissions invalidates all their active sessions
- Self-registration is disabled; only Admins can create Users
- A **Task** has exactly one **Assignee** (a User) and one **Creator** (the User who created it)
- A **Task** can only be reassigned by the **Creator**, the current **Assignee**, or a **Supervisor** of the Assignee
- **Tasks** can only be updated or archived by the **Creator**, the **Assignee**, or a **Supervisor** of the Assignee
- A **Task** may have zero or more **Task Links**
- A **Task Link** references exactly one Task and one external entity (by type + id). Valid entity types: `prospect`, `job_mandate`, `client`, `client_contract`
- A **Reminder** may be attached to a **Task** (with optional relative offset) or standalone (absolute time only)
- Relative Reminders are only valid when attached to a **Task** that has a deadline
- When a **Task's** deadline changes, all attached relative Reminders recompute their `trigger_at`
- A **Task** may have an optional start date; used for weekly Calendar view
- Every change to a **Task** produces a **Task Event** — an immutable audit trail of who changed what and when
- Every core entity (**Client**, **Client Contract**, **Job Mandate**, **Prospect**) produces **Change Records** on every edit — same pattern as Task Events
- Each entity has its own change record table: `client_events`, `contract_events`, `mandate_events`, `prospect_events`
- A **Notification** belongs to exactly one **User** and optionally references an entity via polymorphic `entity_type` + `entity_id`
- A **Notification** is unread when `read_at` is null; marking it read sets the timestamp
- **Notification types**: `task_assigned`, `task_status_changed`, `task_deadline_approaching`, `reminder_triggered`, `mandate_assigned`, `mandate_completed`, `mandate_stage_changed`, `prospect_stage_changed`, `prospect_hired`, `client_created`, `contract_created`, `contract_assigned`, `system`
- **Notifications** are personal — a User sees only their own
- A **Job Mandate** may have zero or more **Prospects** linked to it (via a **Prospect Mandate** record)
- A **Prospect** may be linked to zero or more **Job Mandates** (via **Prospect Mandate** records)
- A **Prospect Mandate** stores the current stage: `prospect`, `applicant`, `candidate`, `hired`, or `rejected`
- A **Prospect** with no Prospect Mandate records is simply a Prospect (no stage attribute)
- The same **Prospect** can be at different stages for different **Job Mandates**
- Every stage change on a **Prospect Mandate** produces a **Prospect Mandate Event** — immutable record of who changed the stage, when, on which mandate
- The prospect→applicant transition is gated by `prospects: update` (Caller or SC advances after contact)
- The applicant→candidate transition is gated by `job_mandates: verify` (QC verification)
- QC rejection sets the stage to `rejected` with a reason (e.g., `internal_qc`)
- Once a Prospect reaches **candidate** stage, the stage is final — they cannot be rejected or reverted
- A Prospect at **candidate** stage transitions to **hired** when the RM confirms placement (triggered when the client communicates the hire to the RM)
- This transition is gated by `job_mandates: assign` (RM role)
- A **Job Mandate** transitions to `hired` status when the count of `hired` stage Prospect Mandates equals the total headcount across all locations (flat count, no location-specific assignment)
- A **rejected** Prospect can be re-linked to the same Job Mandate; the UI must present a confirmation dialog before re-including in the flow

## Schema Designs

### prospect_mandates
```
id          serial PK
prospect_id text FK → prospects.id
mandate_id  integer FK → job_mandates.id
stage       enum(prospect, applicant, candidate, hired, rejected) NOT NULL default 'prospect'
reason      text (nullable — set when stage = 'rejected')
created_at  timestamp
UNIQUE(prospect_id, mandate_id)
```

### prospect_mandate_events
```
id                  serial PK
prospect_mandate_id integer FK → prospect_mandates.id
old_stage           enum(prospect, applicant, candidate, hired, rejected, null)
new_stage           enum(prospect, applicant, candidate, hired, rejected)
changed_by          text FK → user.id
changed_at          timestamp
```

## Views

### Tasks
- All three views (List, Kanban, Calendar) share a single route with a view-mode toggle (`/tasks?view=list|kanban|calendar`)
- Archived **Tasks** are hidden by default; a "show archived" filter toggle reveals them
- **List view**: Sortable, filterable table using bazza/ui data-table-filter + TanStack Table. Server-side filtering via oRPC. Default sort: deadline ascending (no deadline last).
- **Kanban view**: Columns grouped by **Task Status** (`todo`, `in_progress`, `done`). Cards show title, assignee avatar, deadline badge. No drag-and-drop or manual ordering.
- **Calendar view**: Monthly + weekly with a toggle. Monthly shows tasks on deadline day; weekly shows tasks as blocks from start date to deadline. Tasks with no start date appear as point markers on their deadline day.
- **Detail panel**: Slide-out drawer with editable task fields, **Reminders** section, **Task Links** section, and **Task Event** history timeline.
- **Task creation**: "New Task" button on `/tasks` page. Required fields: title, status (defaults to `todo`). All other fields optional.

### Clients
- **List view**: Sortable, filterable table. Scoped by Supervisor hierarchy (Assignee = self + descendants). Archived clients hidden by default.
- **Detail panel**: Slide-out drawer with client info, contracts list, linked mandates.

### Client Contracts
- Shown within **Client detail panel** as a list.
- **Detail panel**: Contract info, assigned RM, linked mandates, SLA reference.

### Job Mandates
- **List view**: Sortable, filterable table. Scoped by assignee + Supervisor hierarchy. Archived/completed mandates hidden by default.
- **Kanban view**: Columns by **Mandate Status** (`open`, `in_progress`, `hired`, `cancelled`). Cards show title, assignee avatar, client badge.
- **Detail panel**: Mandate info, assigned TL/Caller, linked prospects with their stages, SLA reference.

### Prospects
- **List view**: Sortable, filterable table. Visible to all authenticated users. Archived prospects hidden by default.
- **Detail panel**: Prospect info, linked mandates with stages.

### Team Members
- Admin-only page: list of all users with role, supervisor, status.
- **Detail/Edit**: Role assignment, supervisor assignment, permissions editor (for custom role).

### Dashboard
- **Task stats**: total active, by status, overdue count
- **Recruitment pipeline**: prospects by stage (prospect / applicant / candidate / rejected), total across all mandates
- **Active mandates**: count of mandates in `open` and `in_progress` status, scoped by user's visibility
- **Mandates nearing deadline**: mandates with approaching deadlines
- **Recent stage transitions**: latest prospect→applicant and applicant→candidate transitions
- **Client activity**: recently added clients and contracts, scoped by user's visibility
- **Reminders**: upcoming reminders for the current user
- **Notifications**: unread count and recent notifications

## API

- `client.list` / `client.create` / `client.update` / `client.archive` / `client.getById`
- `clientContract.list` / `clientContract.create` / `clientContract.update` / `clientContract.archive` / `clientContract.getById`
- `jobMandate.list` / `jobMandate.create` / `jobMandate.update` / `jobMandate.assign` / `jobMandate.verify` / `jobMandate.archive` / `jobMandate.getById`
- `prospect.list` / `prospect.create` / `prospect.update` / `prospect.archive`
- `prospectMandate.link` / `prospectMandate.updateStage` / `prospectMandate.unlink` / `prospectMandate.listEvents`
- `task.list` — filtered, paginated list with nested assignee/links/reminders (cursor-based pagination)
- `task.create` — create a task
- `task.update` — update a task (server-side field-level diff → Task Events + reminder recomputation)
- `task.archive` — archive a task
- `task.getById` — fetch a single task with nested relations
- `task.listEvents` — edit log for a task
- `task.addLink` / `task.removeLink` — manage Task Links
- `reminder.create` / `reminder.archive` / `reminder.list` — shared Reminder endpoints (optional `task_id`)
- `notification.list` / `notification.markRead` / `notification.archive`
- `teamMember.list` / `teamMember.create` / `teamMember.update` / `teamMember.archive` / `teamMember.updateRole`
